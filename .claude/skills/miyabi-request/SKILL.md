# Miyabi Request

You are the request intake layer for this app repo.

Convert a rough chat request into a clear GitHub Issue that the shared runner can process.

## When To Use

Use this skill when the user asks for implementation work such as add, fix, implement, remove, refactor, test, or says `miyabi-request`.

Do not use this skill for pure explanation, research, review-only, or status questions. Do not create an Issue for casual conversation.

## Rules

- Do not edit app code as part of this intake step unless the user explicitly asked for direct local edits.
- Do not put secrets, tokens, passwords, or private credentials in the Issue body.
- Use the current GitHub repo unless the user clearly names another repo.
- For cross-repo requests, create one parent Issue first and list likely related repos under `Impact Hints`.
- Do not add `🤖agent-execute` by default. Issue opening starts planning. Use execute labels or `/agent execute` only when the user explicitly asks to run execution and the repo gate supports it.
- If the request is too vague, still create a useful planning Issue unless a security-sensitive or impossible detail is missing.
- Always include `## Tasks`. For a small request, write one executable task. For a large or cross-repo request, split it into 2-6 independent executable tasks.
- `## Tasks` is the decomposition entrypoint for Miyabi automation. Do not only restate the user's wish; write tasks that an agent can actually execute.
- If cross-repo work is possible, include `## ImpactDecision`. If it is not resolved at intake, say planning must resolve it before execution.
- If required repos are already known, name them in `## Tasks`; otherwise keep them in `Impact Hints` and leave `ImpactDecision` as not resolved.
- Do not pretend to know exact files when the repo has not been inspected. Use likely surfaces in `Impact Hints` unless the exact target is known.

## Issue Body

Create the Issue with this structure:

```markdown
## Request

<Rewrite the user's request in 1-3 concrete sentences.>

## Goal

<What should be true when the work is done.>

## Requirements

- <Concrete requirement>
- <Concrete requirement>

## Tasks

- [ ] <Executable task generated by the intake agent. Use one item for a small request.>
- [ ] <For a broad or cross-repo request, split into 2-6 independent tasks.>
- [ ] <For known required repos, write `owner/repo`: executable work.>

## Impact Hints

- Target repo: `<owner/repo or current repo>`
- Related repos: `<known related repos, or unknown>`
- Likely surfaces: `<API/UI/config/workflow/docs/etc.>`

## ImpactDecision

- Status: `<not-resolved-at-intake | ready | blocked>`
- Decision source: `<planning resolver confirms this before execution>`
- Required repos: `<unknown, or owner/repo with reason>`
- Required evidence: `pull-request`, `merge-commit`, `ledger-record` for every required repo

## Completion Criteria

- [ ] <Observable success condition>
- [ ] Existing CI / typecheck passes
- [ ] A PR is opened with a clear summary

## Constraints

- Follow `AGENTS.md`
- Do not include secrets or tokens in code, logs, or comments
- Keep changes scoped to the requested behavior

## Automation

- Source: Miyabi request intake
- Initial mode: planning
- Child issue creation: leave to Miyabi decomposition when `## Tasks` shows multiple independent workstreams
- ImpactDecision resolver: run during planning before execution when cross-repo work is possible
```

## Procedure

1. Resolve the target repo:
   - Run `gh repo view --json nameWithOwner -q .nameWithOwner`.
   - If that fails, ask for the repo instead of guessing.
2. Draft the Issue title:
   - Use `[Task] fix: ...`, `[Task] feat: ...`, `[Task] refactor: ...`, `[Task] test: ...`, or `[Task] docs: ...`.
3. Write the Issue body to a temporary file.
4. Run `gh issue create --repo "<owner/repo>" --title "<title>" --body-file "<body-file>"`.
5. Return the Issue URL and a short summary of what you put into it.

## Output To User

Keep the response short:

- Issue URL
- One sentence describing the normalized request
- Whether it is planning-only or execution was explicitly opened
